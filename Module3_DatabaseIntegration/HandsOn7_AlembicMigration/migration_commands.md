# Database Migration Cheat Sheet & Framework Comparison

**Tools**: Alembic (SQLAlchemy) vs. Django Built-In Migrations  

---

## 🛠️ Alembic Migration Command Reference

| Command | Purpose | Description |
| :--- | :--- | :--- |
| `alembic init alembic` | Initialize Environment | Creates `alembic.ini` configuration and `alembic/` script folder. |
| `alembic revision --autogenerate -m "msg"` | Generate Revision | Compares SQLAlchemy models to DB state and generates diff migration script. |
| `alembic upgrade head` | Apply Migrations | Runs `upgrade()` functions for all unapplied revisions up to the latest. |
| `alembic current` | Check Current Revision | Displays currently applied revision hash stored in `alembic_version` table. |
| `alembic history --verbose` | View Migration History | Prints full migration revision chain with hashes and parent relationships. |
| `alembic downgrade -1` | Roll Back One Revision | Runs `downgrade()` function to revert database schema by one revision step. |
| `alembic downgrade base` | Complete Rollback | Reverts ALL applied revisions back to blank database baseline. |

---

## ⚙️ Django Migrations Command Reference

| Command | Purpose | Description |
| :--- | :--- | :--- |
| `python manage.py makemigrations` | Detect & Generate | Inspects `models.py` changes and creates numbered `.py` migration files. |
| `python manage.py migrate` | Apply Migrations | Applies pending migrations to the database. |
| `python manage.py showmigrations` | View Applied Status | Displays list of migrations marked with `[X]` for applied or `[ ]` for pending. |
| `python manage.py migrate courses 0001` | Rollback to Revision | Reverts app migrations back to specified target revision number (`0001`). |
| `python manage.py migrate courses zero` | Rollback All | Undoes all migrations for specified application. |

---

## 📊 Comparative Analysis: Alembic vs. Django Migrations

1. **Schema Diffing Accuracy**:
   - **Django**: Built directly into framework core; reliably tracks column renames (`AlterField`, `RenameField`).
   - **Alembic**: Excellent at table/column additions/deletions, but requires manual inspection of autogenerated files for column renames or server defaults.
2. **Version Tracking**:
   - **Django**: Uses numbered sequence per app (`0001_initial.py`, `0002_add_field.py`) tracked in `django_migrations` table.
   - **Alembic**: Uses GUID revision hashes (`a1b2c3d4e5f6`) stored in `alembic_version` table.
